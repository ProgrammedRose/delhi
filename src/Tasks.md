# ЛР1:Монады

### Можно использовать
- стандартную библиотеку Scala;
- даже если вы используете библиотечные монады, в блоке 0 вы должны реализовать свои варианты монад;
- трансформеры эффектов, если сможете объяснить суть их раюоты (но лучше без них).

### Нельзя использовать
- императивный стиль;
- библиотеки вроде Cats, Cats Effect, ZIO
- голые эффекты;

### Блок 0. Инфраструктура

Перед решением предметной части варианта необходимо реализовать:

- `trait Monad[M[_]]` с операциями `pure`, `flatMap`, `map`;
- наивный `Reader[Env, A]`;
- наивный `Writer[Log, A]`;
- наивный `State[S, A]`;
- наивный `IO[A]` с маленьким `unsafeRun`.

Технически удобно использовать лог вида `Vector[String]` или `List[String]`.

### Блок 1. Reader

Reader используется для в функциях, которые требуют чтения параметров из конфигурации.

### Блок 2. Writer

Writer используется вместо обычного лога. Можно не использовать его посеместно и заменить на IO, но в паре мест пожалуйста пусть будет.

### Блок 3. State

Если для чего-то хочется использовать глобальную переменную, вместо неё используем State.

### Блок 4. IO

Используем для взаимодействия с пользователем через консоль.

### Замечание по оформлению решения

Во всех вариантах желательно придерживаться одной и той же структуры проекта:

1. отдельно определения монад;
2. отдельно предметные типы и функции;
3. отдельно сценарий `IO`;
4. короткий `README` с инструкцией по запуску.

Цель задания — не просто реализовать набор функций, а показать, что вы умеете:

- выносить конфигурацию в `Reader`;
- делать вычисление объяснимым через `Writer`;
- моделировать изменение мира через `State`;
- отделять описание взаимодействия от исполнения через `IO`.

## Вариант 1. Торговый автомат

Сюжет: автомат продаёт напитки, принимает монеты, выдаёт сдачу и хранит остатки товара.

### Reader

Из окружения берутся:

- цены на товары;
- допустимые номиналы монет;
- максимальная сумма, которую можно держать внесённой;
- правило скидки, например: после 18:00 сок дешевле на 10%.

Реализовать:

- `priceOf(product)`;
- `canAcceptCoin(coin)`;
- `effectivePrice(product, hour)`;
- `calculateChange(inserted, price)`.

### Writer

Добавить функции с логом:

- `explainInsert(coin)`;
- `explainPurchase(product, inserted, hour)`;
- `explainFailure(reason)`.

В логе должно быть видно, как формировалась цена и почему покупка прошла или не прошла.

### State

Состояние содержит:

- остатки товаров;
- внесённую сумму;
- накопленную выручку;
- кассу автомата по монетам.

Реализовать переходы:

- `insertCoin`;
- `selectProduct`;
- `cancelPurchase`;
- `refillProduct`.

### IO

Собрать сценарий:

- показать меню;
- прочитать товар;
- прочитать монету или набор монет;
- выполнить покупку;
- вывести лог и итоговое состояние.


## Вариант 2. Библиотека

Сюжет: библиотека выдаёт книги, принимает возвраты и считает просрочки.

### Reader

Из окружения берутся:

- максимальное число книг на одного читателя;
- срок выдачи в днях;
- штраф за день просрочки;
- список запрещённых к выдаче редких книг.

Реализовать:

- `canBorrow(reader, book)`;
- `dueDate(today)`;
- `fine(daysLate)`;
- `isRestricted(book)`.

### Writer

Нужно логировать:

- решение о выдаче;
- дату возврата;
- факт просрочки;
- размер штрафа.

### State

Состояние:

- каталог с количеством экземпляров;
- выданные книги;
- текущий день.

Переходы:

- `borrowBook`;
- `returnBook`;
- `nextDay`;
- `addCopies`.

### IO

Сценарий:

- ввести имя читателя;
- выбрать команду `borrow` или `return`;
- вывести чек операции.

## Вариант 5. Лифт

Сюжет: лифт ездит между этажами, принимает вызовы и перевозит пассажиров.

### Reader

Из окружения:

- число этажей;
- максимальная вместимость;
- этажи, на которые лифт не ездит;
- стоимость одного шага движения в энергии.

Реализовать:

- `canGoTo(floor)`;
- `canEnter(currentPassengers, n)`;
- `travelCost(from, to)`;
- `distance(from, to)`.

### Writer

Логировать:

- получение вызова;
- остановку на этаже;
- посадку и высадку;
- причину отказа.

### State

Состояние:

- текущий этаж;
- число пассажиров;
- очередь заявок;
- потраченная энергия.

Переходы:

- `callLift`;
- `enterPassengers`;
- `exitPassengers`;
- `stepToNextRequest`.

### IO

Сценарий:

- ввести этаж вызова;
- ввести число пассажиров;
- сделать 1–2 шага симуляции.

### Обязательные тесты

| Кейс | Ожидание |
|---|---|
| Корректный вызов | заявка добавлена |
| Запрещённый этаж | отказ |
| Переполнение | посадка не выполнена |
| Выполнение шага | этаж меняется в нужную сторону |

- `unloadShip`;
- `departShip`;
- `tickHour`.

### IO

Сценарий:

- зарегистрировать корабль;
- выполнить один цикл разгрузки;
- вывести состояние порта.


## Вариант 20. Космическая станция

Сюжет: станция распределяет энергию, следит за кислородом и ремонтирует модули.

### Reader

Из окружения:

- норма кислорода на человека;
- лимиты энергии по модулям;
- стоимость ремонта;
- правило аварийного режима.

Реализовать:

- `oxygenNeed(crewSize)`;
- `energyLimit(module)`;
- `repairCost(module)`;
- `shouldEnterEmergency(oxygen, energy, failures)`.

### Writer

Лог:

- распределение энергии;
- падение ресурсов;
- ремонт;
- переход в аварийный режим.

### State

Состояние:

- энергия;
- кислород;
- исправность модулей;
- список задач экипажа;
- текущий день.

Переходы:

- `allocateEnergy`;
- `consumeOxygen`;
- `repairModule`;
- `nextDay`.

### IO

Сценарий:

- вывести состояние станции;
- выбрать ремонт или перераспределение энергии;
- показать лог дня.
